# Phase 3 Research Workspace

This folder tracks battle-dataset Phase 3 work independently from the app and schema work in Phases 1-2.

The files here are mostly curation artifacts for Historical Battles. Paintings curation lives in `paintings-notes.md` plus the `paintings-batch-*` candidate/audit/script files; buildings remain owned by another agent.

## Current Slice

- `candidates/battles.tsv`: 500 battle selections with draft continent, epoch, and optional casualties fields, seeded from the casualty-list starting point and expanded with priority omissions to the 500-card target.
- `candidates/battle-details.json`: curated battle facts and original glosses promoted into production data.
- `candidates/battle-media-audit.json`: Wikipedia/Wikidata/Commons media audit used for image metadata.
- `dataset-visualizations.html`: unified local visualization report for battles, paintings, and buildings, generated by `scripts/build-dataset-visualization-report.mjs`.
- `battles-notes.md`: curation rules, risk notes, and next actions.
- `scripts/`: repeatable extraction, media audit, and promotion scripts.

## Candidate Status Meanings

- `selected-first-pass`: included in the initial 100-card target list.
- `fixture-present`: already appears in the tiny fixture data and should be reconciled against the final candidate row.
- `needs-verification`: facts, source quality, image license, and attribution still need card-level review before promotion into `data/*.json`.
- `rights-risk`: useful for recognition, but image reuse may be restricted. Replace, omit, or use only if the final product policy allows it.

## Production Status

`data/battles.json` currently contains 500 battle cards generated from the battle research files. All cards have required details, series data, source URLs, image source URLs, image license strings, and attribution. Casualties are present on 294 cards, including 30 side-by-side casualty strings, and intentionally omitted where available figures are not clean enough for display.

## Promotion Checklist

Before a row becomes a production card:

1. Verify title, date, location, creator, and taxonomy values against at least two stable sources where possible.
2. Choose one clear image whose reuse terms are acceptable.
3. Record exact image `sourceUrl`, `license`, and `attribution`.
4. For battles, record `details.casualties` only when the estimate can be scoped clearly as a display string; prefer side-by-side casualty strings over combined totals when the source supports it.
5. For battles, keep `details.combatants` as exactly two opposing display sides, grouping allied participants into one side string.
6. Write an original gloss within the manifest character limit.
7. Ensure all manifest-required fields are non-empty and filter values match `data/datasets.json`.
8. Keep unresolved uncertainty in the dataset notes rather than hiding it in the JSON.

## Source URL Convention

The TSV source columns are starting targets, usually Wikipedia or Commons search pages. They are not final attribution records. Final card JSON should use exact source pages, not search result URLs.
